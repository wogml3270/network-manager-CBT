## 시작점

네트워크관리사 2급 시험을 준비하면서 매일 공부 계획을 관리하고, 문제를 풀고, 해설까지 확인할 수 있는 웹 애플리케이션을 만들고 싶었다.

물론 이미 웹에는 CBT 서비스가 있다. 특히 오래전부터 운영되어 온 기출 기반 CBT 사이트들은 실제로 문제를 풀기에는 충분하다.

다만 사용해보면 오래된 PHP 서비스 특유의 화면과 흐름이 강하게 느껴진다. 기능은 있지만 최신 웹 애플리케이션에서 기대하는 UI/UX와는 거리가 있다. 모바일에서 보기 좋게 풀고, 결과를 자연스럽게 확인하고, 공유까지 이어지는 경험을 기대하면 아쉬운 부분이 있었다.

그래서 처음부터 모든 자격증을 다루는 거대한 CBT 서비스를 만들기보다는, 당장 내가 시험을 치르게 될 네트워크관리사 2급부터 현대적인 형태로 다시 만들어보기로 했다.

처음 떠올린 형태는 꽤 컸다. 오늘의 학습 대시보드, 목표 기간 설정, 일별 계획표, 오답노트, 개념노트, JSON 백업/복원, AI 해설까지 포함한 학습 관리 도구에 가까웠다.

하지만 실제로 내가 가장 자주 사용할 화면을 생각해보면 답은 단순했다.

문제를 고른다. 푼다. 제출한다. 해설을 본다. 다시 푼다.

그래서 최종 방향은 학습 플래너가 아니라 CBT 전용 앱으로 바꿨다.

이번 버전은 네트워크관리사 2급에 집중하지만, 장기적으로는 다른 자격증 CBT도 추가할 수 있는 구조를 염두에 두었다. 문제은행, 과목 필터, 채점, 결과 화면을 분리해두면 나중에 새로운 자격증 데이터와 시험 기준을 붙이는 방식으로 확장할 수 있기 때문이다.

## 기존 CBT 서비스와 다른 방향

기존 CBT 서비스는 이미 존재한다. 그래서 이 프로젝트의 목표는 “세상에 없는 문제풀이 기능을 처음 만든다”가 아니었다.

목표는 더 분명했다.

- 첫 화면에서 바로 CBT를 시작할 수 있을 것
- 화면이 모바일과 데스크톱 모두에서 읽기 좋을 것
- 문제 풀이 중 불필요한 요소가 적을 것
- 결과 화면에서 전체 문항과 해설을 빠르게 복습할 수 있을 것
- 지인에게 결과를 공유하기 쉬울 것
- 추후 다른 자격증 문제은행으로 확장할 수 있을 것

기존 PHP 기반 CBT가 “문제를 제공하는 장소”에 가깝다면, 이 앱은 “반복 학습을 편하게 하는 도구”에 더 가깝게 만들고 싶었다.

## 공식 기출을 그대로 넣지 않기로 한 이유

네트워크관리사 2급 기출 자료는 공식 자료실에서 확인할 수 있다. 하지만 공식 기출 원문을 웹앱에 내장해서 배포하는 것은 저작권과 재배포 관점에서 조심해야 한다.

그래서 이 앱에는 공식 기출 문장을 복제하지 않았다.

대신 ICQA 2급 필기 출제 범위를 기준으로 네 과목을 나누고, 각 과목별로 자체 제작 유사문제 50문항씩 총 200문항을 구성했다.

- 네트워크 일반
- TCP/IP
- NOS
- 네트워크 운용기기

문제 데이터는 `data/questions.json` 하나로 관리한다. 처음에는 문제와 풀이 JSON을 나누는 방식도 생각했지만, 수정과 확장을 고려하면 한 문항 안에 문제, 선택지, 정답, 해설을 함께 두는 편이 더 단순했다.

## AI 해설을 제거한 이유

초기 버전에는 OpenAI API를 호출해서 해설을 생성하는 기능도 있었다.

하지만 실제로 써보니 개인 학습 앱에서 매번 유료 API를 호출하는 구조는 부담이 있었다. 특히 이 앱의 목적은 시험 전 반복 풀이였고, 해설은 매번 새로 생성될 필요가 없었다.

그래서 AI 해설을 완전히 제거했다.

대신 각 문제마다 사람이 읽을 수 있는 상세 해설을 정적 JSON에 포함했다. 이 선택 덕분에 앱은 더 단순해졌고, API 키나 과금 상태와 무관하게 누구나 바로 사용할 수 있게 됐다.

## 기능을 줄이면서 남긴 것

기능을 줄였지만 CBT 경험에 필요한 것은 남겼다.

첫 화면은 바로 CBT 설정 화면이다. 대시보드나 랜딩 페이지 없이 과목과 문항 수만 고르면 시작할 수 있다.

문항 수는 20문항, 30문항, 50문항 중에서 선택한다. 기본 CBT 기준은 50문항, 50분, 60점 이상 합격으로 설정했다.

풀이 화면에는 현재 문항 번호, 남은 시간, 선택지, 이전/다음, 제출 버튼만 둔다. 반복해서 쓰는 도구이기 때문에 화면의 장식보다 흐름이 중요했다.

이 선택은 기존 CBT 서비스에 대한 아쉬움과도 연결된다. 문제 풀이 자체는 복잡한 기능이 필요하지 않지만, 사용자가 반복해서 마주하는 화면은 충분히 정돈되어 있어야 한다. 그래서 설정, 풀이, 결과라는 세 단계가 한눈에 이어지도록 구성했다.

## 랜덤 배치

문제를 반복해서 풀다 보면 번호 위치를 외워버리는 문제가 생긴다.

그래서 매번 CBT를 시작할 때 문제 순서를 무작위로 섞고, 각 문제의 선택지 순서도 다시 섞었다. 이때 중요한 점은 선택지가 섞인 뒤에도 정답 인덱스가 정확히 따라가야 한다는 것이다.

이를 위해 `shuffleQuestionChoices`에서 기존 선택지의 원래 인덱스를 함께 보관한 뒤, 섞인 결과에서 정답 선택지의 새 위치를 다시 계산하도록 했다.

작은 로직이지만 문제 풀이 앱에서는 틀리면 안 되는 핵심 부분이었다.

## 미응답 제출 확인

실제 CBT에서는 한두 문제를 실수로 건너뛰는 일이 생길 수 있다.

그래서 제출 버튼을 눌렀을 때 미응답 문항이 있으면 확인 메시지를 띄우도록 했다.

예를 들어 3번과 8번 문제를 선택하지 않았다면 다음과 같은 문구가 나온다.

```text
3, 8번 문제를 선택하지 않았습니다. 제출하시겠습니까?
```

사용자가 취소하면 첫 번째 미응답 문항으로 이동한다. 확인하면 그대로 결과 화면으로 넘어간다.

## 결과 화면

처음에는 결과 화면에서 오답만 보여주면 충분하다고 생각했다.

하지만 실제 복습에서는 맞힌 문제도 해설을 다시 읽어보는 것이 도움이 된다. 특히 맞히긴 했지만 헷갈렸던 문제는 오답보다 더 중요할 때도 있다.

그래서 최종 결과 화면은 전체 문항을 모두 보여준다.

각 문항에는 다음 정보를 표시한다.

- 정답/오답/미응답 상태
- 과목
- 문제
- 내가 선택한 답
- 정답
- 상세 해설

이렇게 바꾸고 나니 결과 화면이 단순 채점표가 아니라 복습 화면 역할까지 하게 됐다.

## 결과 공유

앱을 Vercel에 배포한 뒤 주변 지인들에게 풀어보게 할 계획이 있었기 때문에 공유 기능도 추가했다.

처음에는 소셜 로그인을 붙일 수 있을지 고민했다. 하지만 목적이 “누가 풀었는지 기록한다”가 아니라 “풀고 나서 카카오톡으로 결과를 공유한다”에 가까웠기 때문에 로그인은 필요하지 않았다.

대신 결과 화면에 공유 기능을 추가했다.

- 카카오톡 공유
- Web Share API
- 링크 복사 fallback

카카오톡 공유는 Kakao JavaScript SDK의 `Kakao.Share.sendDefault()`를 사용했다. Kakao 설정이 없거나 브라우저에서 공유가 실패하면 일반 링크 공유나 클립보드 복사로 자연스럽게 대체된다.

공유 링크는 `/share` 페이지로 연결된다. 이 페이지는 점수, 정답 수, 합격 여부를 요약해서 보여주고, 다시 CBT를 풀 수 있는 버튼을 제공한다.

## 배포 파이프라인

배포는 Vercel을 사용했다.

`main` 브랜치에 push하면 Vercel이 production 배포를 만들고, GitHub Actions가 최신 READY 배포를 찾아 `network-manager-cbt.vercel.app` alias로 연결한다.

이 흐름은 기존 `j-blog`에서 사용하던 alias 동기화 방식을 가져와 프로젝트에 맞게 수정한 것이다.

추가한 구성은 다음과 같다.

- `vercel.json`
- `.github/workflows/vercel-alias-sync.yml`
- `scripts/vercel-sync-production-alias.mjs`
- `.env.example`

환경변수는 GitHub Secrets와 Vercel Environment Variables로 나누어 관리한다.

배포 도메인은 `network-manager-cbt.vercel.app`으로 잡았다. 주변 지인들에게 바로 풀어보게 할 예정이었기 때문에, 배포 주소가 최신 production 배포를 안정적으로 가리키는 것도 중요했다.

## 테스트

문제 풀이 앱에서 가장 무서운 버그는 정답이 어긋나는 것이다.

그래서 단위 테스트에서는 점수 계산보다도 문제 데이터와 선택지 랜덤 배치 검증을 중요하게 봤다.

Vitest로 확인한 내용은 다음과 같다.

- 문제은행이 200문항인지
- 과목별로 50문항씩 들어 있는지
- 모든 문항에 선택지 4개가 있는지
- 정답 인덱스가 선택지 범위 안에 있는지
- 선택지를 섞어도 정답 텍스트가 유지되는지
- 공유 URL 생성과 파싱이 정상인지

Playwright로는 실제 브라우저 흐름을 검증했다.

- 첫 화면 렌더링
- CBT 시작
- 답안 선택
- 미응답 제출 확인
- 결과 화면 표시
- 공유 버튼 노출
- 공유 결과 페이지 렌더링

마지막 검증에서는 다음 명령을 모두 통과시켰다.

```bash
npm run test
npm run lint
npm run build
npm run test:e2e
```

## 정리

이번 프로젝트에서 가장 큰 결정은 기능을 추가하는 것이 아니라 기능을 빼는 것이었다.

학습 플래너, 오답노트, 백업/복원, AI 해설은 모두 유용해 보였지만, 지금 필요한 앱의 핵심은 아니었다. CBT를 빠르게 풀고, 결과와 해설을 확인하고, 필요하면 지인에게 공유하는 흐름만 남겼다.

덕분에 앱은 훨씬 단순해졌고, 배포와 유지보수도 쉬워졌다.

무엇보다 공식 기출 원문을 무리하게 내장하지 않고 자체 유사문제로 방향을 바꾼 점이 좋았다. 개인 학습 도구라도 공개 배포를 생각한다면 데이터 출처와 저작권을 먼저 정리해야 한다는 것을 다시 느꼈다.

작지만 목적이 분명한 도구가 됐다.

이번에는 네트워크관리사 2급만 만들었지만, 구조를 더 다듬으면 정보처리, 리눅스마스터, 컴퓨터활용능력 같은 다른 자격증 CBT도 같은 방식으로 확장할 수 있을 것 같다. 기존 CBT 서비스의 장점은 유지하되, 더 현대적인 화면과 흐름으로 다시 만드는 방향을 계속 실험해보고 싶다.
